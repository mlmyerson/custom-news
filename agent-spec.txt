============================================================
I. VISION SUMMARY
============================================================

Morning Issue Radar is a static-hosted, client-side Single Page Application
that gives the user a fast, balanced, issue-centric overview of the daily news.

Instead of scrolling endlessly or trusting algorithmic feeds, the user sees:

1. A "bubble map" of the most important phrases extracted from today’s headlines.
2. Clicking any bubble searches multiple major news sources for that phrase.
3. The app displays all relevant articles across outlets in a single, neutral view.

This shifts the user from headline-chasing to issue-oriented analysis:
a balanced "morning radar sweep" of what's happening in the world.

============================================================
II. ARCHITECTURAL CONSTRAINTS
============================================================

- Static hosting only (e.g., GitHub Pages)
- Client-side JavaScript (no backend servers)
- External APIs must be browser-queryable or lightly proxied
- Dynamic data is fetched at runtime inside the browser
- API keys handled carefully (proxy optional)
- Routing is handled internally via JS or hash-based routing
- All data transformations, NLP, and rendering are client-side

============================================================
III. OVERALL SYSTEM FLOW
============================================================

1. Fetch headlines from major news organizations (AP, Reuters, BBC, NPR, NYT, Guardian, etc.)
2. Extract top phrases (noun phrases, entities) from these headlines.
3. Weight phrases by frequency across sources.
4. Render a bubble map (size/color by weight/category).
5. On bubble click:
   - Run simultaneous searches across major news APIs.
   - Merge, normalize, deduplicate article results.
6. Render results as a structured list (sortable/filterable).
7. Optional: Generate a summary sidebar via an LLM summarization API.

Each step includes: GOAL, INPUTS, OUTPUTS, and COMPLETION CRITERIA.

------------------------------------------------------------
STEP 1 — SPA Skeleton & Routing
------------------------------------------------------------

GOAL:
Create the minimal SPA structure that loads, displays a placeholder bubble map,
and supports internal routing.

INPUTS:
None (basic boilerplate)

OUTPUTS:
- index.html
- app.js
- styles.css
- A simple state-based or hash-based router

COMPLETION CRITERIA:
- App loads cleanly from GitHub Pages
- Placeholder bubble element clickable
- Click switches to Topic View
- "Back" returns to bubble view
- No console errors

------------------------------------------------------------
STEP 2 — Headline Fetch Module
------------------------------------------------------------

GOAL:
Implement client-side API calls to fetch top headlines from 3+ major news sources.

SOURCES (suggested):
- Guardian Open API
- NYT Top Stories API
- AP News RSS
- NPR RSS
- Reuters RSS

OUTPUT:
Normalized array of headline objects:
{
  title,
  summary,
  source,
  url,
  published_at
}

COMPLETION CRITERIA:
- fetchHeadlines() returns consistent structures
- Individual API failures do not break the module
- At least 30 headlines load per session
- Data cached for the user session
- No duplication within normalized set

------------------------------------------------------------
STEP 3 — Keyword Extraction Module
------------------------------------------------------------

GOAL:
Extract top key phrases from gathered headlines.

METHODS:
- Use compromise.js, wink-nlp, or simple noun/phrase rules.
- Score phrases by frequency.

OUTPUT:
[
  { phrase: "Gaza ceasefire", weight: 42 },
  { phrase: "AI regulation", weight: 21 },
  ...
]

COMPLETION CRITERIA:
- Extracts 10–50 meaningful phrases
- No junk terms ("live", "breaking", etc.)
- Deterministic output for identical input
- Weight reflects cross-source frequency

------------------------------------------------------------
STEP 4 — Bubble Map Renderer
------------------------------------------------------------

GOAL:
Render clickable, weighted bubbles for phrases.

REQUIREMENTS:
- Visual clarity and performance
- Interactive hover and click
- Mobile-friendly

COMPLETION CRITERIA:
- Bubbles sized by weight
- Bubbles colored by category or random palette
- Smooth transitions
- Click transitions to Topic View
- Works with >50 bubbles

------------------------------------------------------------
STEP 5 — Multi-Source Article Search Module
------------------------------------------------------------

GOAL:
Given a phrase, query multiple major news search APIs simultaneously.

SOURCES:
- NewsAPI.org
- Guardian Search API
- NYT Article Search API
- Bing News Search API
- Optional: Reddit search, Hacker News Algolia search

OUTPUT:
Unified article list:
{
  title,
  url,
  source,
  snippet,
  published_at
}

COMPLETION CRITERIA:
- Parallel fetch completed <3s for 3–5 sources
- Deduplication based on URL/title similarity
- Errors from individual APIs logged but not fatal
- Returns 10–100 articles for major topics

------------------------------------------------------------
STEP 6 — Topic View Renderer
------------------------------------------------------------

GOAL:
Render results list cleanly with sorting/filtering.

UI:
- Topic heading
- Sort controls (source, recency)
- Optional filtering by domain or source
- Article list (cards or simple list)

COMPLETION CRITERIA:
- Mobile-friendly layout
- Deterministic sorting
- Handles 100+ items smoothly
- No duplicates rendered
- Back navigation restores bubble map without losing state

------------------------------------------------------------
STEP 7 — Optional Summary Module
------------------------------------------------------------

GOAL:
Generate 3–5 bullet summary of top articles for this phrase.

METHOD:
Use a summarization API or LLM endpoint.

OUTPUT:
Short summary paragraph and/or bullet list.

COMPLETION CRITERIA:
- Summary loads in <2 seconds
- No hallucinated events

------------------------------------------------------------
STEP 8 — Final Polish & Deployment
------------------------------------------------------------

GOAL:
Refine UI, performance, and deploy to GitHub Pages.

COMPLETION CRITERIA:
- Deployed successfully
- All API keys loaded safely (via config or proxy)
- No console errors
- Initial load <1 second
- Responsive and stable across desktop/mobile
